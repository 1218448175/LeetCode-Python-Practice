## [76. 最小覆盖子串](https://leetcode.cn/problems/minimum-window-substring/)

### 困难

给定两个字符串 `s` 和 `t`，长度分别是 `m` 和 `n`，返回 s 中的 **最短窗口 子串**，使得该子串包含 `t` 中的每一个字符（**包括重复字符**）。如果没有这样的子串，返回空字符串 `""`。

测试用例保证答案唯一。

<p><strong>示例 1：</strong></p>
<pre><strong>输入：</strong>s = "ADOBECODEBANC", t = "ABC"
<strong>输出：</strong>"BANC"
<strong>解释：</strong>最小覆盖子串 "BANC" 包含来自字符串 t 的 'A'、'B' 和 'C'。
</pre>

---

## 题解：双哈希映射与“达标种类计数”优化策略

### 1. 核心思路：滑动窗口的精确收缩

本题属于滑动窗口中的“最短区间”问题。核心难点在于：如何高效判断当前窗口是否已经完全覆盖了目标字符串 $T$（包括重复字符）。

- **目标拆解**：我们将问题转化为“字符种类”的匹配。如果 $T$ 中有 3 种字符 `A, B, C`，且分别需要 `1, 2, 1` 个，那么只要窗口内这三种字符的计数**全部达标**，当前窗口就是一个“可行解”。

- **窗口动态平衡**：
  
  - **右指针 `r` 扩张**：寻找可行解，直到窗口覆盖所有 $T$ 中字符。
  
  - **左指针 `l` 收缩**：在保证窗口依然“覆盖”的前提下，尽可能压缩长度以寻找最优解。

---

### 2. 执行逻辑的详细拆解

#### A. 关键变量定义

- `target_cnt`：存储 $T$ 中每个字符的必须需求量。

- `window_cnt`：动态记录当前窗口内相关字符的实有量。

- `kinds`：**逻辑核心**。记录当前窗口内有多少种字符的频次已经“达到或超过”了 `target_cnt` 中的要求。

#### B. 扩张与“加分”逻辑

遍历 $S$，当新字符 `in_char` 进入窗口时：

1. 如果它是目标字符，更新 `window_cnt`。

2. **触发达标**：如果 `window_cnt[in_char]` **恰好等于** `target_cnt[in_char]`，说明该种类字符满足了要求，`kinds` 加 1。

#### C. 收缩与“减分”逻辑

当 `kinds == len(target_cnt)`（所有种类均达标）时，开启 `while` 循环：

1. **记录最优解**：此时窗口是合法的，立即更新最小长度 `min_len` 和起始位置 `ans_left`。

2. **尝试破坏平衡**：准备移除左端字符 `out_char`。

3. **触发失标**：如果 `out_char` 是目标字符，且当前数量**刚好等于**需求量，一旦移除，该种类就不再达标，`kinds` 减 1，从而跳出 `while` 循环继续扩张。

---

### 3. 逻辑亮点：临界值判定（`==` vs `>=`）

代码中判断 `kinds` 增减时使用的是 `==` 而不是 `>=`。

- **为什么要用 `==`？** 这样可以确保 `kinds` 只在字符状态从“不达标”变为“达标”的那一瞬间加 1，或从“达标”变为“不达标”的一瞬间减 1。

- **性能优势**：这种方式避免了每次移动指针都去遍历整个哈希表来比对频率，将判断合法性的复杂度从 $O(字符集大小)$ 降到了 $O(1)$。

---

### 4. 复杂度分析

- **时间复杂度**：$O(M + N)$。$M$ 和 $N$ 分别为 $S$ 和 $T$ 的长度。左右指针均单向移动，每个字符最多被访问两次。

- **空间复杂度**：$O(K)$。$K$ 为字符集的大小（如 ASCII 或 Unicode 字符数），本题中通常为常数级别的哈希表开销。
